## 目标（本 skill 只做规范）

当你需要”结合项目代码 / 某个模块代码 + 需求文档 + 需求评审文档”生成开发文档时，输出必须**结构化、可验收、可落地**。

整个流程分为 **两个阶段**，必须按顺序执行：

1. **阶段一：开发处理逻辑**（章节 1-5）— 产出开发文档
2. **阶段二：需求工作量分析** — 基于阶段一的产出，做需求覆盖核验 + 问题清单 + 工作量评估

各阶段涉及的模板引用：

- **需求修改的目录树**：严格使用 `references/目录树格式.md`
- **整个需求前端实现流程图**：严格使用 `references/流程图.md`
- **需求任务拆解**：严格使用 `references/任务拆解.md`
- **需求工作量分析**：严格使用 `references/需求评审.md`

本 skill **只负责规定开发文档的结构与写法**：不替用户写 PRD、不做无边界发散；所有不确定信息必须显式标注为 `TBD`。

## 何时使用（触发条件）

当用户提出以下诉求之一时启用本规范（满足任意一条即可）：

- 需要**结合代码**输出开发说明/开发文档/技术方案（偏前端实现）
- 用户给了需求文档，并要求“结合项目代码/某个模块代码”生成一份可执行的开发文档
- 需要在文档中明确“改哪些文件、加哪些入口、影响哪些链路”
- 需要把“实现流程（生命周期/接口调用/Vuex/组件渲染）”画成可读流程图
- 需要把需求拆成可执行任务并评估工作量（点数/人日/优先级）

## 输入要求（可不完整，但必须显式标注）

允许信息不全，但必须用 `TBD` 占位，禁止默默省略导致实现分歧。最低输入集合：

- **需求一句话**：要改什么、解决什么问题
- **入口与页面**：从哪里进、影响哪些页面/组件
- **后端/接口信息**：接口名、字段、开关、状态码（未知则 `TBD`）
- **边界与异常**：至少列出 5 类（未知则 `TBD`）
- **改动范围**：至少给出初版“可能涉及目录/模块”（未知则 `TBD`）

## 输出结构（开发文档必须包含的章节）

开发文档必须严格按以下顺序组织（可增补 6/7/8…，但不得缺省 **1-5**）：

- **1、涉及改动的页面入口与接口（按目录树格式）**
  - 必须严格套用：`references/目录树格式.md`
  - 至少包含 2 块（块标题与顺序按模板来）：
    - “需要改动/新增的文件（按目录树）”
    - “接口与数据源（xxx 相关）”
  - 约束：
    - 路径必须可在项目中定位；不确定就写候选路径并标 `TBD`
    - 每个文件必须落一句“改什么/为什么”，禁止只罗列文件名

- **2、任务拆解**
  - 必须严格套用：`references/任务拆解.md`
  - 约束：
    - 每个任务块最少包含：**目标** + **涉及文件/模块** + **验收点** 

- **3、流程图**
  - 必须严格套用：`references/流程图.md`
  - 约束：
    - 必须覆盖端到端关键链路：入口/生命周期 → 数据拉取（并行/串行）→ 状态落地（如 Vuex）→ 渲染分支
    - 必须至少包含 1 条异常/兜底分支（例如：接口失败、空态、开关关闭、字段缺失）
    - 禁止改成 mermaid/图片；必须使用模板的 ASCII 方框风格

- **4、关键实现细节**
  - 目的：把“最容易写错/最容易漏”的实现点写清楚，便于直接开工与 Code Review
  - 最少包含：
    - 关键数据结构/字段口径（新增字段、枚举、状态码、空值语义；不确定写 `TBD`）
    - 关键判断与分支（在什么条件下走哪条链路）
    - 与现有逻辑的兼容方式（如何做到不影响非本需求场景）
    - 埋点/日志（如有）：触发时机、字段、失败判定（未知写 `TBD`）
  - 表述要求：
    - 必须落到“文件 + 模块/函数/生命周期 + 为什么”，禁止泛化词（如“按原逻辑/优化一下/处理一下”）

- **5、注意事项 & 风险点**
  - 至少覆盖：
    - 回归影响面（哪些页面/入口/链路需要回归）
    - 风险点（接口字段不稳定、开关灰度、缓存/并发、兼容旧版本数据、国际化等）
    - 兜底/降级策略（失败时用户看到什么、如何恢复、是否可隐藏功能）
    - 边界情况
  - 所有不确定信息必须 `TBD`，并给出“如何确认/谁确认”

- **6、（可选）补充章节**
  - 允许增加 6/7/8…（例如：测试用例建议、发布策略、数据迁移/灰度方案、性能评估等）
  - 约束：不得替代/省略 1-5；并且补充内容必须能落到可执行动作或可验证结果

### 阶段二：需求工作量分析（在章节 1-5 全部产出后执行）

必须严格套用：`references/需求评审.md`

**前置条件**：阶段一（章节 1-5）已全部产出。如未产出，必须先完成阶段一。

**本阶段做什么**：
- 对照需求文档，逐项核验阶段一的开发文档是否完整覆盖所有需求点
- 识别需求文档中的歧义/模糊点，输出给产品/设计/后端的问题清单
- 基于阶段一的任务拆解，逐项评估工作量（点数），输出工作拆分清单与汇总表

**本阶段不做什么**：
- 不重复阶段一的代码定位/目录树/流程图产出
- 不重新拆解任务，而是在阶段一任务拆解的基础上补充工作量评估

**输出结构**：严格按 `references/需求评审.md` 的"固定输出结构"执行，不可变更

## 强制模板引用（不可改格式）

当需要输出“目录树 / 流程图 / 任务拆解”时：

- **目录树**：复制 `references/目录树格式.md` 的结构与写法，不要自创格式
- **流程图**：复制 `references/流程图.md` 的 ASCII 方框风格与信息密度，不要改成 mermaid/图片
- **任务拆解**：复制 `references/任务拆解.md` 的口径，不要换算口径除非用户明确要求
- **需求工作量分析**：复制 `references/需求评审.md` 的输出结构与工作量汇总表格式，不要自创格式

## 写作约束（保证可读性与可维护性）

- **只写确定的事实**：凡是推测必须标 `TBD/待确认`，并写“需要谁确认/怎么确认”
- **路径必须真实**：目录树中的文件路径必须可在项目中定位；不确定就写候选路径并标 `TBD`
- **改动点必须可追溯**：对每个关键改动，至少落到“文件 + 模块/函数/生命周期 + 为什么”
- **异常必须可观察**：异常分支要写明“用户看到什么/埋点或日志如何判断/如何恢复”
- **避免泛化词**：如“优化一下/处理一下/按原逻辑”这类必须落到具体规则或判断条件

## 最小质量检查（交付前自检）

**阶段一自检：**
- [ ] 文档包含 **1-5** 全部章节，且 **1/2/3** 的格式与 `references` 模板一致
- [ ] 目录树里每个文件都有”改什么/为什么”（一句话也行）
- [ ] 流程图覆盖：入口 → 数据 → 状态 → 渲染，并至少 1 条异常分支
- [ ] 任务拆解每个任务都有：目标 + 验收点 + 工作量口径
- [ ] 所有不确定信息都用 `TBD` 标注，并给出”待确认项列表（如何确认/谁确认）”

**阶段二自检：**
- [ ] 已读取需求文档图片并标记页面归属
- [ ] 已对照需求文档逐项核验开发文档覆盖完整性，遗漏项已列出
- [ ] 问题清单（产品/设计/后端）每条含影响说明
- [ ] 开发工作拆分清单每个子项带点数，每个子模块有小计
- [ ] 工作量评估汇总表已输出，含总点数与总人日
- [ ] 已生成汇总 Markdown 文档并返回路径
